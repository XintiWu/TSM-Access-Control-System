Slide 02 / Problem Statement

為什麼這個系統極具挑戰？

90,000+

超大員工基數

分散於多個廠區、辦公大樓，交班尖峰期瞬間湧入龐大刷卡流量。

< 50ms

毫秒級開門決策

門禁設備必須在極短時間內完成權限校驗與 Anti-passback 判定，拒絕卡頓。

OLAP Query

繁雜的層級報表

主管需要跨廠區、跨部門多維度的考勤稽核，歷史海量事件的分析對傳統 DB 負擔極重。

核心技術衝突點： 高頻寫入與即時分析並存。傳統 RDBMS 面臨大表查詢卡頓，因此我們導入事件驅動搭配 ClickHouse 列式儲存引擎 解決 OLAP 報表難題。
Slide 03 / User Persona & Design Thinking

使用者痛點與設計推導

PERSONA 1: 廠區員工

員工上班刷卡體驗

「上班打卡時間緊迫，要是門禁系統卡個幾秒，後方就會大排長龍，導致我們遲到！」

・Pain Point: 系統反應遲鈍、尖峰時刻排隊焦慮。

・Requirement: 門禁驗證優先追求 極低延遲（Latency First）。

PERSONA 2: 部門主管

主管管理與稽核

「我需要產出準確的出勤與稽核報表，如果資料有落差、延遲，會影響工時合規判定。」

・Pain Point: 大規模報表查詢緩慢、考勤資料落差。

・Requirement: 報表呈現優先追求 極致精準與高分析效能（Accuracy & OLAP Performance First）。

🔑 核心共識：系統必須依此分流成 ⚡ Fast Path (門禁決策) 與 🐢 Slow Path (ClickHouse 考勤分析)。

Slide 07 / Anti-Passback Design

防跟隨與重複進出機制 (Anti-Passback)

狀態機轉換模型 (State Machine)

初始狀態: OUTSIDE ➔ 刷卡進入 ➔ INSIDE (核准)
當前狀態: INSIDE ➔ 再次刷卡進門 ➔ DENIED (拒絕: 重重複進)
當前狀態: INSIDE ➔ 刷卡出門 ➔ OUTSIDE (核准)
當前狀態: OUTSIDE ➔ 刷卡出門 ➔ DENIED (拒絕: 重複出)

Redis 狀態存儲架構 (Fast Memory Schema)

我們避免在每次進出刷卡時都去查詢 ClickHouse 列式資料庫，而是在 Redis 內定義高效率的實體記憶體雜湊： 

# 員工狀態
employee:{id}:state ➔ "INSIDE" | "OUTSIDE"

# 前次門禁事件稽核
employee:{id}:last_door ➔ "door_id_101"
employee:{id}:last_time ➔ "1716611822"
💡 特點： 透過記憶體比對可達成 O(1) 複雜度的防跟隨校驗，並將歷史大數據日誌直接寫往 ClickHouse。
Slide 08 / Event-Driven Buffer & Resilience

異步緩衝與斷連容錯機制

SCENARIO A: 正常批量寫入 正常狀態

當 ClickHouse 資料庫正常連接時，DB Worker 會從 Kafka 批次拉取事件日誌（例如：每積累 1,000 筆或等候 1 秒），以列式儲存最高效的方式一次性寫入（Bulk Insert）。

Access API ➔ 運作正常 (開門 < 50ms)

Kafka Queue ➔ 無訊息堆積 (Lag = 0)

ClickHouse ➔ 進行高效欄位壓縮儲存

SCENARIO B: ClickHouse 故障容錯 斷線故障保護

若 ClickHouse 無預警中斷（如資料庫進行維護中），Access API 依然透過記憶體 Redis 完成開門判定。所有刷卡日誌皆安全暫存在 Kafka 訊息佇列中，員工進出完全不受影響。

Access API ➔ 運作正常 (員工不卡門，體驗無感)

Kafka Queue ➔ 持久化保存所有新事件 (Lag 增加)

ClickHouse ➔ 離線維護中，恢復連線後自動補寫

💡 過濾機制： 呼叫 Report API 時，中介軟體根據主管 manager_id 遞迴拉取出所有子部門 ID，動態組裝為 SQL 在 ClickHouse 進行分組高速掃描。

主管儀表板考勤指標 (Metrics)

  • 個人即時停留時間： 依據 ENTER/EXIT 自動相減，提示異常停留。
  • 出勤率與工時計算： ClickHouse 能在亞秒內對百萬級事件進行聚合計算。
  • 未同步事件提醒： 標示出因為數據庫暫時離線所造成的延遲同步數據。
  • 防跟隨異常告警： 異常連續進入（觸發 Anti-passback）即時列表。
ClickHouse 強項： 面對海量考勤，直接對 Raw Events 做高效率聚合查詢，無懼複雜篩選與多欄度組合分析。
Slide 10 / ClickHouse Table Schema

ClickHouse 資料庫建模 (OLAP)

📊 access_control.inout_events

id UUID

event_time DateTime64(3, 'UTC')

employee_id UUID

door_id UUID

direction Enum8('IN' = 1, 'OUT' = 2)

status Enum8('ALLOW' = 1, 'DENY' = 2)

reason Nullable(String)

source_ip String

card_uid String

org_unit_id UUID

ENGINE = MergeTree()

PARTITION BY toYYYYMM(event_time)

ORDER BY (org_unit_id, event_time, employee_id)

📈 access_control.pre_aggregated_reports

report_date Date

org_unit_id UUID

department_id UInt32

total_entries SimpleAggregateFunction(sum, UInt64)

total_exits SimpleAggregateFunction(sum, UInt64)

unique_employees AggregateFunction(uniq, UUID)

ENGINE = AggregatingMergeTree()

PRIMARY KEY (org_unit_id, report_date)

ORDER BY (org_unit_id, report_date)

💡 ClickHouse 架構精髓： 欄位式儲存不使用強外鍵關聯（FK）。表結構採取輕度冗餘設計，透過 `LowCardinality` 優化重複字串，並利用 `PARTITION BY` 進行按月物理分割，大幅收斂查詢掃描範圍。
Slide 11 / Cloud Native Deployment

Kubernetes 雲原生部署拓撲

Stateless Deployments

負責門禁與報表邏輯的 API 服務，均採用無狀態（Stateless）設計，部署於多 Pod 中。

k8s: access-api, admin-api, report-api
Stateful Components

Kafka 佇列、Redis 快取與 **ClickHouse (OLAP)** 屬於有狀態元件，利用 StatefulSet 與 PV 實作。

StatefulSet: ClickHouse, Kafka, Redis
Security & Monitoring

專為叢集安全與可觀測性設計的系統服務，配置於特權或常駐 DaemonSet 中。

Prometheus, Grafana, ingress-nginx

Slide 16 / Live Demo Workflow

現場系統 Demo 流程與驗證劇本

STEP 1: 基礎流程與反潛回

透過 make verify-pipeline 模擬刷卡。

  • 正常刷卡：卡號 9999，回傳 ALLOW。
  • 反潛回 (Anti-Passback)：連續刷卡，第二次回傳 DENY。
命令：curl access-api:8080/inout
STEP 2: 封禁快取失效 (Cache)

驗證 Admin 封禁與 Redis 同步更新。

  • 呼叫 admin-api Banned 員工。
  • 透過 Cache Invalidation 異步清除 Redis 狀態。
  • 再次刷卡，秒級反應回傳 DENY。
命令：curl admin-api:8081/ban
STEP 3: 斷線容錯與異步同步

ClickHouse 離線高可用容錯。

  • 暫停 worker 容器 模擬斷網。
  • 繼續刷卡：照常回傳 ALLOW（Redis 決策）。
  • 重啟 worker，事件從 Kafka 補入 ClickHouse。
命令：docker stop aggregation-worker
STEP 4: 報表聚合與可觀測性

多維分析與指標觀測。

  • 進入 report-api，查看 ClickHouse 預聚合視圖 秒出考勤報表。
  • 開啟 Grafana 看 QPS、Kafka Lag 與 ClickHouse 效能指標。
URL：localhost:8082/reports
Slide 17 / Engineering Trade-offs

面臨挑戰與工程權衡 (Trade-offs)

1. Latency vs Consistency

我們優先保證極低延遲的開門決策（⚡ Fast Path）。因此捨棄了寫入 DB 的強一致性，採用 Redis + Kafka 進行最終一致性，極大提升了高峰期員工卡打卡體驗。

🔍 權衡結果：極致體驗 ➔ 最終一致性。
2. OLAP Analysis vs Row Persistence

由於考勤資料每日高達數百萬筆，傳統 RDBMS 在大表關聯查詢容易癱瘓。我們引入 **ClickHouse** 作為列式存儲，將大表多維聚合查詢加速至「亞秒級」，以此支撐主管層級報表。

🔍 權衡結果：列式分析資料庫 ➔ 高效查詢。
3. Simplicity vs High Scalability

雖然單體架構在初期開發極其簡單，但考量到 90K 員工規模，我們仍不辭繁複地拆分出 Access API、Kafka 佇列、以及 Redis Cache 分層架構，大幅增強了未來的橫向彈性與水平擴容。

🔍 權衡結果：捨棄簡易性 ➔ 換取超高擴展。
Slide 18 / Summary & Conclusion

實踐雲原生：以架構賦能業務

本系統透過事件驅動（Event-Driven）架構成功將門禁判定與複雜報表分析完美解耦。 ⚡ Fast Path 運用 Redis 高效運算和 Kafka 異步緩衝，全面滿足毫秒級通行體驗，即使在資料庫故障期間也能無縫運轉。 🐢 Slow Path 則結合 ClickHouse 為主管呈現高精度且極速的層級出勤報表。

結合 Kubernetes HPA 水平擴展機制以及全方位的監控面板，系統具備極佳的自動化彈性，隨時準備應對高峰與尖峰流量的嚴酷挑戰。
