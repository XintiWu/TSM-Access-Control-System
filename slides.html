Slide 02 / User Pain Points & System Challenge

使用者痛點 → 系統核心困難

Persona 1: Employee

員工：刷卡不能慢

「上班打卡時間緊迫，如果門禁卡住幾秒，後方就會排隊，甚至造成遲到。」

Pain Point: 尖峰時段排隊、重複刷卡、擔心因系統延遲影響出勤。

Requirement: 門禁驗證必須低延遲且可預期，開門決策不能被慢查詢阻塞。

Persona 2: Manager

主管：報表不能錯

「我需要準確的出勤與稽核報表，如果資料遺漏，會影響人力配置與決策。」

Pain Point: 報表查詢慢、資料延遲或不完整，會降低主管對系統的信任。

Requirement: 歷史資料要完整、可追溯，並支援跨部門與組織樹分析。

Fast Need 開門決策要在毫秒內完成，避免排隊。
Core Conflict 高頻寫入與複雜 OLAP 查詢不能互相拖累。
Accurate Need 報表可以短暫延遲，但資料不能遺漏。
Slide 03 / Why Fast Path & Slow Path

為什麼要拆成 Fast Path 與 Slow Path？

Option A

單一路徑：API 直接寫 DB

開發最簡單，但刷卡、寫入、報表查詢全部擠在同一條路上。

報表查詢或 DB 寫入卡住時，會直接拖慢開門決策。
Fast Path

門禁決策：Latency First

只回答一件事：這次刷卡能不能開門？ 使用 Redis 做權限與 Anti-passback 判定，避免慢查詢。

不等 DB、不等報表、不做 aggregation，先回應刷卡設備。
Slow Path

紀錄與報表：Accuracy First

Kafka 先吸收事件，再由 Worker 寫入 ClickHouse，支援部門、時間、門禁點等多維度分析。

報表可以短暫延遲，但 raw data 不能遺漏。
設計結論： Fast Path 保護使用者刷卡體驗，Slow Path 保護資料完整性與報表效能；兩者解耦後，報表尖峰不會反向影響開門。

Slide 05 / API Boundary Design

三個 API 各自負責什麼？

Access API

Door Decision Gateway

Input: CardID, DoorID, Direction, Timestamp

責任: 驗證權限、黑名單、Anti-passback 狀態。

Output: Open / Deny，並非同步送出 raw event。

不負責：報表查詢、歷史聚合、複雜 DB Join。

Report API

Analytics & Query Layer

Input: User / Dept / Time Range / Report Type

責任: 讀取 ClickHouse 或 Report Cache，套用主管權限過濾。

Output: 個人、部門、組織樹、歷史考勤報表。

不參與：開門判定，避免報表查詢影響刷卡延遲。

Admin API

Policy Management

Input: 使用者、部門、門禁規則、黑名單設定。

責任: 管理權限資料與組織樹，發出 permission change event。

Output: 更新 DB，觸發 Redis cache invalidation。

讓權限更新同步到 Fast Path，但不直接處理刷卡流量。
API 拆分原則： Access API 專注低延遲，Report API 專注查詢與分析，Admin API 專注規則與權限管理；每個 API 都有清楚邊界，才能獨立擴容與維護。
Slide 06 / Access API Internal Logic

Access API：50ms 內到底做了什麼？

Critical Path Target: < 50ms
1
Receive Swipe

接收 CardID / DoorID / Direction / Timestamp

2
Validate Request

檢查格式、設備來源、必要欄位與基本風險

3
Redis Permission Check

O(1) 查權限、黑名單、門禁規則，不查主資料庫

4
Anti-passback State

確認上一筆進出方向，避免重複進門或跟隨入侵

5
Return Decision + Publish Event

先回傳 Open / Deny，再把 raw event 非同步送到 Kafka

為了低延遲，我們刻意不做

不做 DB Join 或複雜查詢

不等 ClickHouse insert 完成

不在請求內產生報表

不讓 Worker 狀態阻塞開門

可靠性設計

即使後端分析資料庫暫時變慢，Access API 仍可依靠 Redis cache 完成判定；事件先進 Kafka，後續由 Worker 重試與補寫。

Fast Path 的成功標準：門能即時且正確地開或拒絕。
Slide 07 / Anti-Passback Design

防跟隨與重複進出機制 (Anti-Passback)

狀態機轉換模型 (State Machine)

OUT
廠外狀態 (OUTSIDE)
刷卡動作 ⬌
IN
廠內狀態 (INSIDE)
當前狀態: OUTSIDE
Swipe IN (進入) ALLOW ✅ ➔ IN
Swipe OUT (離開) DENY ❌ 重複出
當前狀態: INSIDE
Swipe OUT (離開) ALLOW ✅ ➔ OUT
Swipe IN (進入) DENY ❌ 尾隨重進

Redis 狀態存儲架構 (Fast Memory Schema)

我們避免在每次進出刷卡時都去查詢 ClickHouse 列式資料庫，而是在 Redis 內定義高效率的實體記憶體雜湊： 

# 權限快取 (僅儲存被封禁名單)
perm:denied:{userId} ➔ "DENY"

# 反潛回狀態
passback:{userId} ➔ "IN" | "OUT"

# 門禁設備健康狀態
door:status:{doorId} ➔ "ONLINE" | "OFFLINE"
特點： 透過記憶體比對可達成 O(1) 複雜度的防跟隨校驗，並將歷史大數據日誌直接寫往 ClickHouse。

Kafka 的角色

不是單純存資料，而是把開門延遲與資料持久化速度分開。ClickHouse 慢或暫停時，事件仍可保留等待補寫。

Worker 的角色

以 batch 方式消費事件、轉換欄位、處理 retry，避免每次刷卡都同步寫入分析資料庫。

Report 的角色

保留 raw events 供稽核，並建立常用彙總資料，讓主管 dashboard 不需即時計算所有歷史紀錄。

設計結論：Fast Path 只要把事件送進 Kafka 就能結束；Slow Path 則負責把事件可靠地轉成報表資料，達到削峰、容錯與可稽核。

階層式部門權限與考勤統計報表

A 廠區廠長 ➔ 權限：A 廠區、所有子研發/製造部門
Team A1 主管 ➔ 權限：Team A1 成員 (無法看 A2)

過濾機制： 呼叫 Report API 時，中介軟體根據主管 manager_id 遞迴拉取出所有子部門 ID，動態組裝為 SQL 在 ClickHouse 進行分組高速掃描。

主管儀表板考勤指標 (Metrics)

  • 個人即時停留時間： 依據 ENTER/EXIT 自動相減，提示異常停留。
  • 出勤率與工時計算： ClickHouse 能在亞秒內對百萬級事件進行聚合計算。
  • 未同步事件提醒： 標示出因為數據庫暫時離線所造成的延遲同步數據。
  • 防跟隨異常告警： 異常連續進入（觸發 Anti-passback）即時列表。
ClickHouse 強項： 面對海量考勤，直接對 Raw Events 做高效率聚合查詢，無懼複雜篩選與多欄度組合分析。

專為叢集安全與可觀測性設計的系統服務，配置於特權或常駐 DaemonSet 中。

Prometheus, Grafana, ingress-nginx

GCP (GKE) 上的 HPA 實作與配置

Slide 12 / GCP Cloud Integration

在 GCP (GKE) 上實作 HPA 與雲端整合

GKE Cluster Autoscaler

當 HPA 橫向擴展 Pod 時，若 GKE 叢集資源不足，自動調度 Cluster Autoscaler 在 asia-east1 自動擴充 GKE 工作節點 VM，實現「Pod 擴展 ➔ 節點擴充」雙重彈性擴展。

自動優化成本，離峰時收縮節點。

GCP Workload Identity

捨棄傳統 IAM Key 檔案，透過 GCP Workload Identity 將 GKE ServiceAccount 安全連結至 GCP IAM 服務帳戶（GSA）。實現最小權限原則，無痛存取 GCP 託管資源。

免密鑰認證，免除金鑰外洩風險。

Secret Manager & ESO

敏感資訊（如 ClickHouse 密碼、API Key）集中管理於 GCP Secret Manager，GKE 內部利用 External Secrets Operator (ESO) 異步同步至 Kubernetes Secret 資源，供 Pod 安全掛載。

集中管理金鑰，支援動態輪替。

Slide 15 / Failure Scenario & Consistency Strategy

故障情境與一致性策略

ClickHouse Down

門禁不受影響；Kafka 繼續保留事件，Worker 暫停或 retry。Report API 顯示「資料延遲」。

Impact: Reporting only
Kafka Lag High

Access API 仍可回應；報表落後。Prometheus 偵測 consumer lag，擴增 Worker 或告警。

Action: Scale workers
Redis Unavailable

Fast Path 最高風險點。需 Redis replica / Sentinel / Cluster；短時間 fallback local cache，但安全性下降。

Risk: Door decision
Permission DB Down

既有員工仍可由 Redis cache 判斷；Admin 更新暫停，恢復後由 permission-events 重新同步。

Degrade: No rule update
Door Decision權限與 Anti-passback 必須即時正確。
Event Persistence事件寫入採最終一致，可延遲不可遺漏。
Report Result報表允許延遲，但必須揭露同步狀態。
Slide 16 / Technology Trade-off Matrix

技術選型不是堆工具，而是取捨

Component Chosen Alternatives Why We Choose It Trade-off
API Language Go Java / Node.js / Python 低延遲、高併發、單一 binary，適合 Access API 與 Worker。 框架生態較精簡，需管理 goroutine 與 timeout。
Event Queue Kafka RabbitMQ / Redis Streams 高吞吐、可保留事件、適合 append-only access log。 維運與 partition 設計較複雜。
Fast Cache Redis Local memory / DB only O(1) 查詢權限、黑名單與 Anti-passback 狀態。 需要 Cache Invalidation 與 HA 設計。
OLAP Storage ClickHouse MariaDB / PostgreSQL / ES 列式儲存與聚合查詢快，適合歷史出勤報表。 不適合高頻交易更新與複雜 OLTP。
Orchestration Kubernetes HPA VM manual scaling 尖峰刷卡與 Worker lag 可水平擴容。 部署初期複雜度較高。
Decision principle: Fast Path 選低延遲與高可用；Slow Path 選高吞吐、可追溯與高聚合效能。
