為什麼這個系統極具挑戰？

90,000+

超大員工基數

分散於多個廠區、辦公大樓，交班尖峰期瞬間湧入龐大刷卡流量。

< 50ms

毫秒級開門決策

門禁設備必須在極短時間內完成權限校驗與 Anti-passback 判定，拒絕卡頓。

OLAP Query

繁雜的層級報表

主管需要跨廠區、跨部門多維度的考勤稽核，歷史海量事件的分析對傳統 DB 負擔極重。

核心技術衝突點： 高頻寫入與即時分析並存。傳統 RDBMS 面臨大表查詢卡頓，因此我們導入事件驅動搭配 ClickHouse 列式儲存引擎 解決 OLAP 報表難題。
Slide 03 / User Persona & Design Thinking

使用者痛點與設計推導

PERSONA 1: 廠區員工

員工上班刷卡體驗

「上班打卡時間緊迫，要是門禁系統卡個幾秒，後方就會大排長龍，導致我們遲到！」

・Pain Point: 系統反應遲鈍、尖峰時刻排隊焦慮。

・Requirement: 門禁驗證優先追求 極低延遲（Latency First）。

PERSONA 2: 部門主管

主管管理與稽核

「我需要產出準確的出勤與稽核報表，如果資料有落差、延遲，會影響工時合規判定。」

・Pain Point: 大規模報表查詢緩慢、考勤資料落差。

・Requirement: 報表呈現優先追求 極致精準與高分析效能（Accuracy & OLAP Performance First）。

核心共識：系統必須依此分流成 Fast Path (門禁決策) 與 Slow Path (ClickHouse 考勤分析)。
Slide 04 / Design Decision Overview

從使用者矛盾推導架構決策

Employee Side

門禁決策不能等

上班尖峰時段，員工只在意刷卡後能不能立即進門；任何 DB 查詢、報表計算或寫入延遲，都會直接變成排隊與遲到風險。

Design Goal：Latency First
Access decision must be isolated.

一套系統，兩種相反負載

Fast Path： write-heavy、latency-sensitive、只做開門判斷。
Slow Path： read-heavy、accuracy-sensitive、處理報表與稽核。
先解耦，再選技術
Manager Side

報表資料不能錯

主管需要跨部門、跨時間、跨廠區的歷史考勤分析；報表可以短暫延遲，但不能遺漏資料，否則會影響人力配置與稽核可信度。

Design Goal：Accuracy First
Reporting must be optimized for OLAP.
核心決策：門禁流程只留下必要的即時判斷，事件寫入與報表分析改由 Kafka、Worker 與 ClickHouse 在 Slow Path 處理。這樣尖峰刷卡不會被報表查詢拖慢，報表也不用被開門延遲限制。

Slide 06 / Why Fast Path & Slow Path

為什麼要拆成 Fast Path 與 Slow Path？

Option A

單一路徑：API 直接寫 DB

開發最簡單，但刷卡、寫入、報表查詢全部擠在同一條路上。

報表查詢或 DB 寫入卡住時，會直接拖慢開門決策。
Fast Path

門禁決策：Latency First

只回答一件事：這次刷卡能不能開門？ 使用 Redis 做權限與 Anti-passback 判定，避免慢查詢。

不等 DB、不等報表、不做 aggregation，先回應刷卡設備。
Slow Path

紀錄與報表：Accuracy First

Kafka 先吸收事件，再由 Worker 寫入 ClickHouse，支援部門、時間、門禁點等多維度分析。

報表可以短暫延遲，但 raw data 不能遺漏。
設計結論： Fast Path 保護使用者刷卡體驗，Slow Path 保護資料完整性與報表效能；兩者解耦後，報表尖峰不會反向影響開門。

Slide 08 / API Boundary Design

三個 API 各自負責什麼？

Access API

Door Decision Gateway

Input: CardID, DoorID, Direction, Timestamp

責任: 驗證權限、黑名單、Anti-passback 狀態。

Output: Open / Deny，並非同步送出 raw event。

不負責：報表查詢、歷史聚合、複雜 DB Join。

Report API

Analytics & Query Layer

Input: User / Dept / Time Range / Report Type

責任: 讀取 ClickHouse 或 Report Cache，套用主管權限過濾。

Output: 個人、部門、組織樹、歷史考勤報表。

不參與：開門判定，避免報表查詢影響刷卡延遲。

Admin API

Policy Management

Input: 使用者、部門、門禁規則、黑名單設定。

責任: 管理權限資料與組織樹，發出 permission change event。

Output: 更新 DB，觸發 Redis cache invalidation。

讓權限更新同步到 Fast Path，但不直接處理刷卡流量。
API 拆分原則： Access API 專注低延遲，Report API 專注查詢與分析，Admin API 專注規則與權限管理；每個 API 都有清楚邊界，才能獨立擴容與維護。

Slide 11 / Anti-Passback Design

防跟隨與重複進出機制 (Anti-Passback)

狀態機轉換模型 (State Machine)

OUT
廠外狀態 (OUTSIDE)
刷卡動作 ⬌
IN
廠內狀態 (INSIDE)
當前狀態: OUTSIDE
Swipe IN (進入) ALLOW ✅ ➔ IN
Swipe OUT (離開) DENY ❌ 重複出
當前狀態: INSIDE
Swipe OUT (離開) ALLOW ✅ ➔ OUT
Swipe IN (進入) DENY ❌ 尾隨重進

Redis 狀態存儲架構 (Fast Memory Schema)

我們避免在每次進出刷卡時都去查詢 ClickHouse 列式資料庫，而是在 Redis 內定義高效率的實體記憶體雜湊： 

# 權限快取 (僅儲存被封禁名單)
perm:denied:{userId} ➔ "DENY"

# 反潛回狀態
passback:{userId} ➔ "IN" | "OUT"

# 門禁設備健康狀態
door:status:{doorId} ➔ "ONLINE" | "OFFLINE"
特點： 透過記憶體比對可達成 O(1) 複雜度的防跟隨校驗，並將歷史大數據日誌直接寫往 ClickHouse。
Slide 12 / Why Kafka

為什麼需要 Kafka？隔離尖峰與故障

WITHOUT KAFKA

Access API 直接寫 DB

同步等待：每次刷卡都要等 DB insert 成功，DB 慢就會拖慢開門。

尖峰塞車：交班時間大量事件同時寫入，connection pool 可能被耗盡。

故障連鎖：DB 暫時不可用時，門禁體驗也會被牽連。

Badge → API → DB write → Door response
WITH KAFKA

Kafka 作為事件緩衝層

快速回應：Access API 判斷完成後即可回傳，不等待分析資料庫。

流量削峰：尖峰事件先進 topic，再由 Worker 依能力批次消費。

故障隔離：ClickHouse 或 DB 短暫異常時，事件仍可保留在 queue 中等待恢復。

Badge → API → Kafka → Worker → ClickHouse
Decision：Kafka 的角色不是單純「存資料」，而是把 開門延遲 與 資料持久化速度 分開，讓 Fast Path 不被 Slow Path 拖慢。
Slide 13 / Event-Driven Buffer & Resilience

異步緩衝與斷連容錯機制

SCENARIO A: 正常批量寫入 正常狀態

當 ClickHouse 資料庫正常連接時，Aggregation Worker 會從 Kafka 批次拉取事件日誌（例如：每積累 1,000 筆或等候 1 秒），以列式儲存最高效的方式一次性寫入（Bulk Insert）。

Access API ➔ 運作正常 (開門 < 50ms)

Kafka Queue ➔ 無訊息堆積 (Lag = 0)

ClickHouse ➔ 進行高效欄位壓縮儲存

SCENARIO B: ClickHouse 故障容錯 斷線故障保護

若 ClickHouse 無預警中斷（如資料庫進行維護中），Access API 依然透過記憶體 Redis 完成開門判定。所有刷卡日誌皆安全暫存在 Kafka 訊息佇列中，員工進出完全不受影響。

Access API ➔ 運作正常 (員工不卡門)

Kafka Queue ➔ 持久化保存所有新事件 (Lag 增加)

ClickHouse ➔ 離線維護中，恢復連線後自動補寫

Slide 16 / Why ClickHouse

為什麼選 ClickHouse 做報表分析？

REPORTING WORKLOAD

報表查詢不是一般交易查詢

查某段時間內大量事件。

依部門、員工、門禁點、日期做 group by。

高階主管需要跨組織樹的歷史彙總。

資料多為 append-only log，很少逐筆更新。

WHY CLICKHOUSE

列式 OLAP 更符合資料型態

Columnar Storage：查詢只讀需要的欄位，適合大量聚合。

Compression：大量重複欄位如 DoorID、DeptID 壓縮效果佳。

Partition / Sort Key：依日期與組織維度加速歷史查詢。

Materialized View：常用報表可以預先聚合，提升 Dashboard 回應速度。

Decision：MariaDB / PostgreSQL 適合權限、使用者、組織等交易資料；ClickHouse 則負責海量門禁事件的分析查詢。把 OLTP 與 OLAP 分開，避免歷史報表拖慢核心系統。
Slide 17 / ClickHouse Table Schema

ClickHouse 資料庫建模 (OLAP)

access_control.inout_events

id UUID

event_time DateTime64(3, 'UTC')

employee_id UUID

door_id UUID

direction Enum8('IN' = 1, 'OUT' = 2)

status Enum8('ALLOW' = 1, 'DENY' = 2)

reason Nullable(String)

source_ip String

card_uid String

ENGINE = MergeTree()

PARTITION BY toYYYYMM(event_time)

ORDER BY (org_unit_id, event_time, employee_id)

access_control.pre_aggregated_reports

report_date Date

org_unit_id UUID

department_id UInt32

total_entries SimpleAggregateFunction(sum, UInt64)

total_exits SimpleAggregateFunction(sum, UInt64)

unique_employees AggregateFunction(uniq, UUID)

ENGINE = AggregatingMergeTree()

PRIMARY KEY (org_unit_id, report_date)

ORDER BY (org_unit_id, report_date)

ClickHouse 架構精髓： 欄位式儲存不使用強外鍵關聯（FK）。表結構採取輕度冗餘設計，透過 `LowCardinality` 優化重複字串，並利用 `PARTITION BY` 進行按月物理分割，大幅收斂查詢掃描範圍。
Slide 18 / Why Go

為什麼後端選用 Go？

Low Latency API

Go 是編譯型語言，執行效率穩定，適合 Access API 這種需要快速回應的服務。

High Concurrency

goroutine 適合處理大量 I/O：Redis 查詢、Kafka publish / consume、Worker 背景處理。

Cloud Native Fit

單一 binary、Docker image 較小、啟動快，適合 Kubernetes HPA 快速 scale out。

適合放在 Go 的服務

Access API、Admin API、Report API、Kafka Consumer Worker、Cache Invalidation Worker。

Trade-off

Go 的框架生態不像 Java Spring 完整，錯誤處理較繁瑣，也需要團隊理解 context timeout 與 goroutine 管理。

Slide 19 / Technology Trade-off Matrix

技術選型不是堆工具，而是取捨

Component Chosen Alternatives Why We Choose It Trade-off
API Language Go Java / Node.js / Python 低延遲、高併發、單一 binary，適合 Access API 與 Worker。 框架生態較精簡，需管理 goroutine 與 timeout。
Event Queue Kafka RabbitMQ / Redis Streams 高吞吐、可保留事件、適合 append-only access log。 維運與 partition 設計較複雜。
Fast Cache Redis Local memory / DB only O(1) 查詢權限、黑名單與 Anti-passback 狀態。 需要 Cache Invalidation 與 HA 設計。
OLAP Storage ClickHouse MariaDB / PostgreSQL / ES 列式儲存與聚合查詢快，適合歷史出勤報表。 不適合高頻交易更新與複雜 OLTP。
Orchestration Kubernetes HPA VM manual scaling 尖峰刷卡與 Worker lag 可水平擴容。 部署初期複雜度較高。
Decision principle: Fast Path 選低延遲與高可用；Slow Path 選高吞吐、可追溯與高聚合效能。

Slide 21 / Failure Scenario

故障情境分析：壞掉時系統如何降級？

ClickHouse Down

門禁不受影響；Kafka 繼續保留事件，Worker 暫停或 retry。Report API 顯示「資料延遲」。

Impact: Reporting only
Kafka Lag High

Access API 仍可回應；報表落後。Prometheus 偵測 consumer lag，擴增 Worker 或告警。

Action: Scale workers
Redis Unavailable

Fast Path 最高風險點。需 Redis replica / Sentinel / Cluster；短時間 fallback local cache，但安全性下降。

Risk: Door decision
Permission DB Down

既有員工仍可由 Redis cache 判斷；Admin 更新暫停，恢復後由 permission-events 重新同步。

Degrade: No rule update
Design goal: 報表服務可以延遲，資料寫入可以排隊，但實體開門決策不應被 Slow Path 或資料庫故障拖住。

Slide 27 / Live Demo Workflow

現場系統 Demo 流程與驗證劇本

STEP 1: 基礎流程與反潛回

透過 make verify-pipeline 模擬刷卡。

  • 正常刷卡：卡號 9999，回傳 ALLOW。
  • 反潛回 (Anti-Passback)：連續刷卡，第二次回傳 DENY。
命令：curl access-api:8080/inout
STEP 2: 封禁快取失效 (Cache)

驗證 Admin 封禁與 Redis 同步更新。

  • 呼叫 admin-api Banned 員工。
  • 透過 Cache Invalidation 異步清除 Redis 狀態。
  • 再次刷卡，秒級反應回傳 DENY。
命令：curl admin-api:8081/ban
STEP 3: 斷線容錯與異步同步

ClickHouse 離線高可用容錯。

  • 暫停 worker 容器 模擬斷網。
  • 繼續刷卡：照常回傳 ALLOW（Redis 決策）。
  • 重啟 worker，事件從 Kafka 補入 ClickHouse。
命令：docker stop worker
STEP 4: 報表與可觀測性

多維分析與指標觀測。

  • 進入 report-api，查看 ClickHouse 預聚合視圖 秒出考勤報表。
  • 開啟 Grafana 看 QPS、Kafka Lag 與 ClickHouse 效能指標。
URL：localhost:8082/reports
Slide 28 / Engineering Trade-offs

面臨挑戰與工程權衡 (Trade-offs)

1. Latency vs Consistency

我們優先保證極低延遲的開門決策（Fast Path）。因此捨棄了寫入 DB 的強一致性，採用 Redis + Kafka 進行最終一致性，極大提升了高峰期員工打卡體驗。

權衡結果：極致體驗 ➔ 最終一致性。
2. OLAP Analysis vs Row Persistence

由於考勤資料每日高達數百萬筆，傳統 RDBMS 在大表關聯查詢容易癱瘓。我們引入 **ClickHouse** 作為列式存儲，將大表多維聚合查詢加速至「亞秒級」，以此支撐主管層級報表。

權衡結果：列式分析資料庫 ➔ 高效查詢。
3. Simplicity vs High Scalability

雖然單體架構在初期開發極其簡單，但考量到 90K 員工規模，我們仍不辭繁複地拆分出 Access API、Kafka 佇列、以及 Redis Cache 分層架構，大幅增強了未來的橫向彈性與水平擴容。

權衡結果：捨棄簡易性 ➔ 換取超高擴展。
Slide 29 / Summary & Conclusion

實踐雲原生：以架構賦能業務

本系統透過事件驅動（Event-Driven）架構成功將門禁判定與複雜報表分析完美解耦。 Fast Path 運用 Redis 高效運算和 Kafka 異步緩衝，全面滿足毫秒級通行體驗，即使在資料庫故障期間也能無縫運轉。 Slow Path 則結合 ClickHouse 為主管呈現高精度且極速的層級出勤報表。

結合 Kubernetes HPA 水平擴展機制以及全方位的監控面板，系統具備極佳的自動化彈性，隨時準備應對高峰與尖峰流量的嚴酷挑戰。
